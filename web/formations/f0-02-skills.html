Le concept

Un skill, c'est une recette réutilisable

Concrètement : un dossier contenant un fichier d'instructions (SKILL.md) et, au besoin, des fichiers d'appui — gabarits, exemples, scripts. Claude le découvre et l'utilise au bon moment.

🪄

Invocation automatique

Claude lit la description de chaque skill et décide lui-même de l'utiliser quand la demande correspond. Pas besoin d'y penser.

⌨️

Invocation manuelle

Chaque skill devient aussi une commande : taper /nom-du-skill le déclenche explicitement. On garde le contrôle.

📦

Avec ses ressources

Un skill peut embarquer des gabarits, des documents de référence ou des scripts. Tout ce qu'il faut pour bien faire la tâche, au même endroit.

📁

Skills personnels

~/.claude/skills/ — disponibles dans tous vos projets. Idéal pour vos méthodes de travail à vous.

🤝

Skills de projet

projet/.claude/skills/ — versionnés avec le projet et partagés avec toute l'équipe. Idéal pour les standards communs.

Sous le capot

Anatomie d'un skill

Un dossier dont le nom devient le nom du skill, un fichier SKILL.md obligatoire, et des ressources optionnelles.

# Structure d'un skill
.claude/skills/
└── compte-rendu/ ← le nom du skill
    ├── SKILL.md ← obligatoire
    ├── gabarit.md ← optionnel
    └── exemples/ ← optionnel
        └── exemple-1.md
# Deux façons de le déclencher

> Fais le compte-rendu du meeting
→ invocation automatique (description)

> /compte-rendu notes-meeting.txt
→ invocation manuelle (slash command)
.claude/skills/compte-rendu/SKILL.md
---
name: compte-rendu
description: Rédige un compte-rendu
  de réunion structuré à partir de
  notes brutes. Utiliser dès que
  l'utilisateur mentionne un meeting,
  une rencontre, des notes ou un
  compte-rendu à produire.
---

# Compte-rendu de réunion

## Étapes
1. Lire les notes fournies
2. Utiliser le gabarit gabarit.md
3. Structurer : décisions, actions,
   responsables, échéances
4. Ton neutre et factuel, en français

## Format de sortie
Un fichier Markdown nommé
CR-AAAA-MM-JJ.md
L'en-tête YAML (frontmatter)

Les métadonnées entre les ---. Deux champs essentiels : name et description. C'est la « fiche d'identité » du skill.

description Le champ crucial

C'est la seule chose que Claude lit en permanence. Elle doit dire quoi le skill fait ET quand l'utiliser. C'est elle qui déclenche — ou pas — le skill.

Les instructions (le corps)

Des étapes concrètes, dans l'ordre : le processus à suivre, les fichiers d'appui à consulter, le format attendu. Du procédural, pas de la théorie.

Les ressources d'appui

Gabarits, exemples, scripts placés à côté du SKILL.md. Claude ne les charge que quand le skill s'active — le contexte reste léger.

Le secret du déclenchement

Tout se joue dans la description

Le réflexe naturel de Claude est de sous-utiliser les skills. Une description vague ne déclenchera rien. La bonne recette : décrire la tâche + énumérer les situations et mots-clés qui doivent l'activer.

✕ Description faible

« Aide à rédiger des comptes-rendus. »

Trop vague. Claude ne fera pas le lien avec « résume-moi le meeting d'hier » ou « qu'est-ce qui a été décidé ? ». Le skill dormira dans son dossier.

✓ Description efficace

« Rédige un compte-rendu de réunion structuré à partir de notes brutes. Utiliser dès que l'utilisateur mentionne un meeting, une rencontre, des notes de réunion, des décisions à documenter ou un compte-rendu — même s'il ne demande pas explicitement un "compte-rendu". »

Quoi + quand + les formulations réelles des utilisateurs. Un peu insistante, et c'est voulu.

Le bon outil au bon endroit

Quoi mettre dans un skill (et quoi mettre ailleurs)

Règle simple : CLAUDE.md pour ce qui est toujours vrai, un skill pour un savoir-faire ponctuel qu'on veut réutiliser.

📌

CLAUDE.md

Le contexte permanent du projet : conventions, commandes, interdits. Chargé à chaque session, donc court.

🧰

Skill

Une procédure : « comment produire un compte-rendu », « comment préparer une release ». Chargé seulement quand c'est pertinent.

💬

Le prompt du moment

Les détails du cas présent : quel fichier, quelle réunion, quelle urgence. Ce qui change à chaque fois reste dans la conversation.

L'accélérateur

Skill Creator : le skill qui crée des skills

Plutôt que d'écrire vos skills à la main, le plugin skill-creator transforme Claude en assistant de création : il vous interviewe, rédige, teste et affine le skill avec vous.

Cerner l'intention

Claude pose les bonnes questions : que doit faire le skill ? Quand doit-il se déclencher ? Quel format de sortie ?

Rédiger le brouillon

Il écrit le SKILL.md complet — frontmatter, étapes, ressources — selon les bonnes pratiques.

Tester

Il propose des prompts de test et les exécute pour vérifier que le skill se déclenche et produit le bon résultat.

Optimiser

Il affine la description pour un meilleur déclenchement et itère avec vous jusqu'à satisfaction.

🎬 DÉMO LIVE

On passe au partage d'écran

  1. Montrer l'arborescence : ~/.claude/skills/ et .claude/skills/ dans un projet.
  2. Ouvrir un SKILL.md existant et relire ensemble frontmatter + instructions.
  3. Lancer skill-creator : « Je veux créer un skill qui fait [tâche du quotidien de l'équipe] ».
  4. Dérouler l'interview, regarder le skill se générer, puis le tester dans une nouvelle session.

L'essentiel

À retenir du module 2

  1. Un skill = un dossier + un SKILL.md. Le nom du dossier devient le nom du skill.
  2. La description est reine. Quoi + quand + mots-clés déclencheurs. Soyez précis, voire insistant.
  3. Du procédural, pas de la doc. Des étapes actionnables ; les gabarits et références en fichiers d'appui.
  4. Skill Creator fait le gros du travail. Interview, rédaction, tests : commencez toujours par lui.
