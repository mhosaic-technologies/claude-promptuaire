La politique d'équipe : encadrer l'usage sans le freiner

La règle numéro un

Un CLAUDE.md versionné est un babillard, pas un coffre-fort

Tout ce qui entre dans un fichier versionné est distribué à chaque personne qui clone le dépôt, relu par Claude à chaque session, et conservé pour toujours dans l'historique git. Le test simple : si vous ne l'afficheriez pas au babillard de la salle d'équipe, ça n'entre pas dans le CLAUDE.md.

✓ Sa place est dans le CLAUDE.md

  • Les conventions de code et de commits de l'équipe
  • Les commandes du projet (tests, lint, build)
  • L'architecture et le vocabulaire métier
  • Les interdits d'usage (« ne jamais toucher /legacy »)
  • Les pointeurs vers la documentation interne publique

✕ N'y entre JAMAIS

  • Clés d'API, jetons, mots de passe — même « temporaires »
  • Chaînes de connexion et URL d'infrastructures sensibles
  • Données clients ou renseignements personnels
  • Secrets d'affaires, projets non annoncés
  • Tout ce qu'on ne mettrait pas dans un courriel à toute l'équipe
Les secrets ont déjà une maison.

Variables d'environnement, gestionnaire de secrets, fichier .env ignoré par git — et, rappel du module 6.1, une règle Read(./.env) en deny pour que Claude n'y touche jamais. Un secret qui a transité par git est considéré comme compromis : on le révoque, on ne l'efface pas.

Avant d'installer

Un plugin tiers, ça se passe en entrevue

Installer un plugin ou un serveur MCP, c'est remettre un trousseau de clés à du code écrit par quelqu'un d'autre : il peut ajouter des commandes, des hooks, des accès à des services externes. On ne remet pas les clés sans vérifier les références — voici la grille d'entrevue.

QuestionCe qu'on vérifieSignal d'alarme
Qui le publie ?Auteur identifiable, dépôt public actif, éditeur connu ou communauté vivante.Auteur anonyme, dépôt abandonné, copie d'un projet populaire.
Que fait son code ?Lire les commandes, hooks et scripts avant d'installer — pas seulement le README.Code obscurci, binaires sans source, scripts téléchargés à l'exécution.
Où vont les données ?Les appels réseau : quels domaines sont contactés, avec quel contenu.Envois vers des domaines inconnus ou sans rapport avec la fonction annoncée.
Quels accès exige-t-il ?Les outils et permissions demandés sont proportionnels à ce que le plugin fait.Accès larges (tout le disque, tout le web) sans justification claire.
Qui l'approuve chez nous ?Une révision consignée par la ou le tech lead avant tout déploiement d'équipe.Installé « juste pour essayer » directement sur un poste de travail.
Claude peut faire la pré-lecture pour vous.

« Lis ce plugin et dis-moi ce que ses hooks et scripts font réellement, appels réseau compris » : en quelques minutes, on obtient un portrait honnête. Le verdict, lui, reste humain.

Passer à l'échelle

Une source officielle plutôt que dix copies qui divergent

Quand chaque personne bricole sa propre configuration, la politique s'effrite en silence. Un plugin d'organisation regroupe les skills, commandes, hooks et réglages approuvés — une seule version, maintenue à un seul endroit, mise à jour pour tout le monde en même temps. Comme la trousse de premiers soins : un contenu standard, un responsable, pas d'improvisation.

Tech lead / champion
Maintient le catalogue

Tient le plugin d'organisation à jour, révise les plugins tiers selon la grille, publie les mises à jour et documente ce qui est approuvé.

Gestionnaire
Porte la politique

Fait vivre la charte d'usage, arbitre les demandes d'exception, suit les coûts et s'assure que la conformité et la TI sont dans la boucle.

Chaque équipier
Utilise et signale

Travaille avec le catalogue approuvé, propose des ajouts plutôt que d'installer en douce, et remonte les frictions — une règle contournée est une règle mal calibrée.

Confidentialité

Savoir ce qui sort, décider ce qui reste

Impossible de trancher ce qui est acceptable sans savoir ce qui circule. Le principe est simple : tout ce que Claude lit pour travailler est envoyé à l'API pour produire ses réponses — le reste ne bouge pas de vos machines.

📤

Ce qui sort

Vos demandes, les fichiers que Claude lit, le contexte chargé (CLAUDE.md, skills) : tout cela transite vers l'API d'Anthropic pour générer les réponses.

🗄️

Ce qui reste

Le reste du dépôt — Claude ne lit que le nécessaire — et tout ce que vos règles deny mettent hors d'atteinte : secrets, dossiers sensibles, données clients.

📜

Ce que dit l'entente

Les offres commerciales n'utilisent pas vos données pour entraîner les modèles par défaut. Faites valider l'entente qui couvre vos comptes par la conformité — une fois, par écrit.

La question à poser pour chaque dépôt : « si ce contenu sortait, quel serait l'impact ? »

Réponse « aucun » : on fonce. Réponse « gênant » : on encadre avec des deny. Réponse « grave » (données réglementées, code sous entente stricte) : ce dépôt attend une décision explicite de la conformité avant d'ouvrir Claude dedans.

Atelier de clôture

La charte d'usage : le code de la route de l'équipe

Pas un document juridique de quarante pages : une seule page qui dit ce qui est permis, ce qui est encadré, ce qui est interdit — et qui décide. Rédigée avec l'équipe plutôt qu'imposée d'en haut : c'est ce qui fait qu'elle sera suivie.

Nommer les interdits

Secrets, données clients, dépôts exclus : la liste courte et non négociable, formulée sans ambiguïté.

Cadrer les approbations

Qui révise les plugins, qui modifie le settings.json du projet, qui tranche les exceptions.

Outiller la règle

Traduire la charte en deny/allow, hooks et plugin d'organisation. Une règle outillée survit aux départs.

Réviser à date fixe

L'outil évolue vite : relire la charte chaque trimestre, retirer ce qui bloque pour rien, ajouter ce qui manque.

🎬 DÉMO LIVE

Atelier : réviser un vrai plugin, amorcer la charte

  1. Ouvrir un plugin réel proposé par la salle et dérouler la grille d'entrevue, question par question.
  2. Demander à Claude : « résume ce que font les hooks et scripts de ce plugin, appels réseau compris ».
  3. Rendre le verdict ensemble : installer, installer avec restrictions, ou refuser — et noter pourquoi.
  4. Amorcer la charte au tableau : recueillir trois interdits et une règle d'approbation. Le brouillon repart avec l'équipe.

L'essentiel

À retenir du module 6.2

  1. Un fichier versionné est public. Jamais de secrets ni de données sensibles dans un CLAUDE.md — le test du babillard.
  2. Réviser avant d'installer. Plugins et MCP tiers passent l'entrevue : auteur, code, appels réseau, accès demandés.
  3. Standardiser par un plugin d'organisation. Une source officielle avec un responsable nommé, plutôt que dix configurations qui divergent.
  4. La charte s'outille. Une règle écrite s'oublie ; traduite en permissions et hooks, elle s'applique toute seule.
