Le modèle de permissions : garder le contrôle sans freiner l'équipe

Claude Code est puissant parce qu'il agit — il modifie des fichiers, lance des commandes. La bonne nouvelle : chacun de ces gestes passe par un système de permissions qu'on peut ouvrir, resserrer ou verrouiller, poste par poste ou pour toute l'organisation.

~40 minutes Prérequis : formation F0
projet-web — claude
> Nettoie le dossier de build

⏺ Bash(rm -rf dist/)

✻ Cette commande exige votre accord :
  ❯ 1. Autoriser une fois

Le comportement de base

Prudent par défaut : lire librement, agir sur approbation

Pensez à un consultant externe en première semaine : il peut consulter le dossier autant qu'il veut, mais rien ne change et rien ne sort sans votre signature. C'est exactement la posture de Claude Code à l'installation.

👀

Lire, sans demander

Explorer les fichiers du projet, chercher dans le code, comprendre l'architecture. La lecture seule ne modifie rien : elle est permise d'emblée.

✋

Agir, avec votre accord

Modifier un fichier, lancer une commande, consulter le web : Claude s'arrête et demande. Vous voyez le geste exact avant qu'il se produise.

🧱

Verrouiller, si on veut

Certaines actions peuvent être interdites d'avance : plus de question, un refus net. C'est la partie qui intéresse la gouvernance.

D'où viennent les « Toujours autoriser » ?

Chaque fois que quelqu'un répond « ne plus me demander », la règle s'écrit dans un fichier settings.json. Sans encadrement, la politique de sécurité de l'équipe se construit donc toute seule, un clic à la fois. Ce module sert justement à reprendre la main sur ces fichiers.

Le mécanisme

Trois listes qui décident à votre place

Le modèle fonctionne comme un trousseau de clés qu'on remet à Claude : allow pour les clés confiées, ask pour les portes à double vérification, deny pour les portes condamnées. Chaque règle nomme un outil et, au besoin, un motif précis.

✅

allow

Les actions exécutées sans interruption. On y met les gestes fréquents et sans danger — jamais des accès larges « pour aller plus vite ».

🔔

ask

Claude peut le faire, mais chaque occurrence exige un oui explicite. Idéal pour les gestes lourds de conséquences, comme pousser du code.

⛔

deny

Refus automatique, sans question posée. Un deny l'emporte sur tout allow, peu importe où il est déclaré.

Exemple de règleCe qu'elle couvreListe conseillée
Bash(npm run test:*)Les scripts de test du projet — et rien d'autre. Le motif après la parenthèse restreint la commande.allow
WebFetch(domain:docs.anthropic.com)La consultation web, mais limitée à ce domaine.allow
Bash(git push:*)Pousser vers le dépôt distant : possible, mais confirmé à chaque fois.ask
Read(./.env)La lecture du fichier de secrets du projet. Claude n'y touchera jamais.deny
Bash(curl:*)Toute commande curl — une façon de couper les envois réseau improvisés.deny
L'ordre de préséance est fixe : deny > ask > allow.

Impossible de contourner un interdit en ajoutant une autorisation plus loin. C'est ce qui rend le modèle prévisible — et auditable : la commande /permissions affiche en tout temps les règles actives et leur provenance.

La gouvernance

Qui décide ? Trois étages, et l'organisation a le dernier mot

Les mêmes règles peuvent être posées à trois niveaux, comme les règlements d'un immeuble : le locataire décore son logement, la copropriété fixe les règles communes, et le code du bâtiment s'impose à tout le monde.

managed-settings.json
Organisation — non négociable

Déployé par la TI sur les postes (hors du dossier de l'utilisateur). Les règles qu'il contient ne peuvent être contournées par personne : c'est l'outil de conformité de l'entreprise.

.claude/settings.json
Projet — versionné et partagé

Commité dans le dépôt, donc revu en merge request comme du code. C'est ici que vit la politique de l'équipe : quiconque clone le projet en hérite automatiquement.

.claude/settings.local.json
Projet — personnel, non versionné

Ignoré par git. C'est là que s'accumulent les « toujours autoriser » individuels — pratique, mais invisible pour l'équipe : à surveiller.

~/.claude/settings.json
Utilisateur — valable partout

Les préférences personnelles qui suivent la personne dans tous ses projets. Utile pour ses propres garde-fous, sans effet sur les collègues.

En cas de conflit, l'étage le plus haut gagne.

Les réglages d'organisation priment sur ceux du projet, qui priment sur ceux de l'utilisateur. Et rappel : un deny posé à n'importe quel étage bloque, même si un allow existe ailleurs.

Sous le capot

Toute la politique de l'équipe tient dans un fichier

Voici un .claude/settings.json de projet réaliste : les tests roulent sans friction, les gestes irréversibles se confirment, les secrets sont hors d'atteinte.

.claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(npm run test:*)",
      "Bash(npm run lint)",
      "WebFetch(domain:docs.anthropic.com)"
    ],
    "ask": [
      "Bash(git push:*)"
    ],
    "deny": [
      "Read(./.env)",
      "Read(./secrets/**)",
      "Bash(curl:*)"
    ]
  }
}
allow — les clés confiées

Des motifs précis (npm run test:*), jamais un outil au complet. On autorise des gestes connus, pas des catégories entières.

ask — la double vérification

Pousser du code reste possible, mais chaque git push exige un oui explicite. Le réflexe pour tout geste difficile à annuler.

deny Gagne toujours

Refus net, sans question : les secrets (.env, dossier secrets/) et les envois réseau improvisés sont hors jeu, peu importe ce que les autres listes permettent.

Un fichier versionné = une politique revue

Ce fichier vit dans le dépôt : chaque changement passe en revue de code, chaque poste qui clone le projet hérite des mêmes règles. La gouvernance devient traçable.

🎬 DÉMO LIVE

Des permissions restrictives en action

  1. Ouvrir /permissions dans une session Claude Code : montrer les règles actives et l'étage d'où vient chacune.
  2. Ajouter Read(./.env) à la liste deny du .claude/settings.json du projet.
  3. Demander « affiche la configuration de la base de données » — observer le refus immédiat, sans même une question.
  4. Ajouter Bash(npm run test:*) en allow, relancer les tests et constater qu'ils roulent maintenant sans interruption.

Rappel de F2, angle gouvernance

Les hooks : des garde-fous qui s'exécutent tout seuls

Les listes filtrent par nom d'outil et par motif. Pour une politique plus fine, les hooks — vus en F2 — prennent le relais : de vrais scripts déclenchés autour de chaque action de Claude. Une consigne écrite peut s'oublier ; un hook, comme un détecteur de fumée, ne dort jamais.

🚦

Bloquer avant le geste

Un hook PreToolUse examine chaque action avant son exécution et peut la refuser selon vos propres critères : commande qui touche la production, fichier hors périmètre, mot de passe détecté dans le texte.

🧾

Tracer pour auditer

Un hook peut journaliser chaque commande exécutée : qui, quoi, quand, dans quel projet. Le jour où la conformité pose des questions, la réponse existe déjà.

Permissions et hooks se complètent — et se versionnent ensemble.

Les listes couvrent 90 % des besoins avec zéro code ; les hooks appliquent les règles que les listes ne savent pas exprimer. Déclarés dans le même .claude/settings.json, ils voyagent avec le projet et s'imposent à toute l'équipe.

L'essentiel

À retenir du module 6.1

  1. Lecture libre, action sous accord. Par défaut, Claude explore le projet mais ne modifie rien sans votre approbation explicite.
  2. Trois listes : allow, ask, deny. Et la préséance est fixe — un deny l'emporte toujours, peu importe où il est déclaré.
  3. La portée fait la gouvernance. Utilisateur, projet versionné, organisation : les réglages déployés par la TI ont le dernier mot.
  4. Les hooks appliquent sans exception. Une consigne s'oublie ; un script versionné bloque et journalise à tout coup.
